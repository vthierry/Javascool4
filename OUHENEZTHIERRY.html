Modifications structurelles:
  1. [développement de jvs4] Tous les sources de javascool csore, builder et web sont regroupés dans un projet netbean unique dans svn:javascool/work
    1. avec un build.xml unique(et des fichier ant inclus) qui génère tous les jars et site web
    2. ant a besoin de son extension ant-contrib http://ant-contrib.sourceforge.net pour fonctionner
  2. [organisation du code] Le code java est complètement documenté en français
    1. (notre façon de coder doit aussi servir d'exemple puisque nos sources sont visibles)
      1. chaque html donne accès au source en lecture: donc il faut joliment coder les mecs !!
    2. les fichiers package.html / package.txt sont des mémos de description
    3. le code a été annoté (pour l'instant sous forme de commentaires) dans la perspective de le préprocessuer avec atp
    4. Hints: les packages à une classe, les doublons, les classes orphelines, etc... sont virés 
    5. Tout le traitement des interruptions a été revu car on arrivait pas à les récupérer
      1. Il était géré à partir de reportApplicationBug ou de java.util.logging.* ou pas, selon les cas, des exceptions étaient masquées, d'utres génbéraient elles mêmes des NullPointerException !!
    6. Beaucoup de constructions static ont été remplacés par des constructions propres d'objets voir Console.java
    7. J'ai essayé de modulariser au maximum les objets (à commencer par ceux issus de Jvs3.3bis!) et
      1. et ai vraiment séparé le gui du moteur de JavaScool 
      2. arrêter de faire des appels vers des variables/fonctions statiques entre toutes les classes qui rend le code incontrolable
      3. bien séparer les mécanismes algorithmiques des méthodes de glue avec les obkets graphiques (voir HtmlDisplay par exemple)

Régressions ou ruptures par rapport à la Alpha-fin de juillet:
  1. L'architecture de JavaScool Core est basée sur deux objets
    1. Desktop qui définir les functions d'interaction avec l'interface graphique de JavaScool.
    2. Engine qui implémente les mécanismes de (compilation, exécution, proglet, ..) JavaScool.
  2. Les éléments : ToolBar (ex JVSToolBar) HtmlDisplay StartStopButton ont été sorti de javascool.gui car ils peuvent être utilisés pour faire des proglets.
  3. Les routines Macros.getRessource/Macros.getUserRessource ont été fondue avec lex toURl devenu: Macros.getResourceURL()
  4. javascoolbuilder a perdu (provisoirement) son progress bar
